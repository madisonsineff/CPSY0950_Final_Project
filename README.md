## Project Overview
Our GamePython app is a collection of interactive mini-games inspired by the online message game, GamePigeon. The project includes a main hub screen where users can choose from multiple games--Connect Four, Anagrams, Filler, and Cup Pong. The purpose of this project was to practice Python programming, interactive game design, Pygame, project organization, and debugging across multiple files. We wanted to incorporate aspects of vibecoding and learn to efficiently and morally utilize AI, via Cursor, into our work while also developing our technical framworks and ability to debug and rationalize through code. Seeing as our first project was through MatLab PsychToolbox and was experiment-based, this final project is quite different and allowed us to develop skills we have not previously worked on--speciifcally concerning game development. We hope you have fun playing!

## Requirements
To run this project, you need:
- Python 3
- Pygame or Pygame-ce

No additional datasets or data files are required for this project. All images and audio files needed to run the games are included in the project folder.

## Challenges We Encountered

While developing this project, we faced a variety of challenges. Although the project was not always technically challenging in the strictest sense because we used AI tools such as Cursor to help with portions of the code, it still required a lot of hands-on editing, testing, and debugging. We did not want to fully rely on AI-generated code without understanding it, so a major part of the process was reading through the code, making changes ourselves, fixing errors, and making sure each part of the project actually worked exactly the way we envisioned.

One challenge was adjusting the visual layout of the games. Many parts of the project, such as buttons, game boards, text, backgrounds, and interactive elements, required precise spacing and positioning. This often involved a lot of trial and error because we had to launch the app repeatedly to see whether the spacing looked correct. Even small changes to coordinates, font sizes, or image placement could affect the appearance of the whole game screen, so this process was time-consuming.

Another major challenge was getting the code to run properly on our (specifically Pearl's) computers. This was also a challenge encountered in our previous group project. Before we could even test the games, we had to figure out how to install the correct version of Python and Pygame. It took around two hours to troubleshoot because Cursor was not allowing pygame to function properly and multple python installments were made and deleted until it eventally worked.

Cup Pong was easily the most difficult games to develop because, unlike the other games, it required us to think about a more three-dimensional style of interaction. While games like Connect Four, Anagrams, and Filler are basically based on two-dimensional layouts, Cup Pong had to simulate depth, aiming, movement, and object placement in a way that felt visually believable to the player. A major challenge was making the game look and feel like the ball was traveling toward cups in space rather than just moving across a flat screen. This required careful adjustment of the cup locations and overall visual perspective. Even small changes in coordinates could make the ball look like it was missing a cup, floating awkwardly, or not matching the intended target. The physics of Cup Pong also made it more complicated than the other mini-games because we had to actually incorporate moving objects and consider how the ball should move, how aiming should work, and how the game should detect whether the ball landed in the cup, etc. One way we were able to overcome this challenge was switching to a mechanism that was more computer friendly as opposed to what the mobile app typically uses. Original Cup Pong is played on a phone, where players will flick the screen with their finger to launch the ball. We originally tried to replicate this with a drag forward motion of the mouse -- however, we ran into a lot of problems with trying to track this motion and generate accurate results of where the ball would go. As a result, we decided to instead, change the mechanism to a slingshot drag back motion which was much easier to add conditions for differing power as well as control lateral motion aiming before the ball was launched.  Because this game depended so much on precise locations and timing, it involved a lot of trial and error. We had to repeatedly run the app, test the ball movement, adjust positions, and then test again.

Overall, even with AI assistance, this project required significant time, effort, and problem-solving. We spent many hours testing, editing, troubleshooting, and making design decisions. The process helped us learn how to use AI as a support tool rather than a replacement for our own understanding, while also giving us more experience with Python, Pygame, debugging, and interactive game development.
